# 版本记录

## 2026-07-24 (external file write confirmation)

- `write` / `edit` 文件工具访问当前 permission profile 未明确允许且未被 deny 的路径时，固定触发确认，不再受工具 `defaultAction = "allow"` 影响。
- 修正 `:workspace_roots "."` 对外部绝对路径的匹配边界，只匹配当前 workspace 内路径。
- 外部写入确认使用 `external-file-write-boundary` 规则 ID，复用现有“允许一次”交互和审计，并新增“本 session 始终允许当前文件 / 当前文件夹”两个 session 授权选项。
- 文件级 session 授权只复用相同 active profile、相同工具和相同目标路径摘要；文件夹级 session 授权复用同一文件夹下后续外部 `write/edit` 调用，但不跨文件夹。
- 多目标分布在不同文件夹时不提供文件夹级 session 授权，避免一次确认扩大到多个目录。
- 补充 `test/policy.test.ts` 和 `test/extension.test.ts` 覆盖外部写入确认、deny 优先、允许一次重复询问、文件级授权、文件夹级授权和跨目录隔离。
- 发布版本提升到 `0.1.7`。

## 2026-07-21 (srt spawnHook)

- 修复 `wrapWithSrt = true` 在 Pi Agent 0.80.x 中只写 SRT settings/audit、但没有真正改写 bash 执行命令的问题。
- `tool_call` hook 改为只承担审批、确认、阻断和 `srtBinary` 存在性检查，不再依赖修改 `event.input.command`。
- 高版本 Pi Agent 暴露 `createBashToolDefinition` 时，pi-perm 注册同名 `bash` 覆盖工具，并通过 bash `spawnHook` 在实际 spawn 前生成 SRT settings、返回 `srt --settings ... <command>` 包装命令。
- `srt_settings` 审计事件只在 spawnHook 实际写 settings 后记录，避免 audit 日志制造“已包裹执行”的误导。
- 兼容不支持 bash `spawnHook` 的 Pi Agent：保留审批/阻断能力，启动提示显示 `bash spawnHook unavailable`，但该环境下无法保证 SRT 包装真正进入 bash 子进程。
- 记录本机 SRT 运行时限制：即使命令已被包裹，若 macOS sandbox profile 拒绝 SRT 创建 `/tmp/claude/` mux Unix socket，`srt` 仍可能以 `listen EPERM` 启动失败。
- 补充 `test/index.test.ts` 覆盖 extension 入口注册 bash override；调整 `test/extension.test.ts` 覆盖 spawnHook 包装路径。
- 发布版本提升到 `0.1.6`。

## 2026-07-07 (permission profiles)

- **BREAKING**：旧 `profiles.<name>.sandbox.*` 权限配置模式不再兼容；文件系统和网络权限统一迁移到 `permissions.<name>`。
- 新增 `activePermissionProfile` 与 named permission profiles，使用 `read` / `write` / `deny` 描述文件系统权限，用 `network.enabled`、`domains` 和 `unixSockets` 描述网络边界。
- 默认 `workspace` 权限 profile 允许 workspace 内普通读写，保护 `.git`、`.codex`、`.agents` 和 `.env` 等敏感路径，网络默认关闭。
- `bash` 工具在权限边界内默认自动放行，命中危险 `tools.bash.operations`、禁网联网命令、deny 路径或高风险能力时仍确认或阻断。
- `read` / `write` / `edit` 文件工具改为按 effective permission profile 判定，优先级为 `deny > write > read`。
- SRT settings 改为从 effective permission profile 生成文件系统、网络和 Unix socket 边界。
- `/pi-perm list` 和 `/pi-perm use <profile>` 切换 permission profiles；`pi_perm_policy` 摘要展示当前 active permission profile。
- 更新 `defaults/base.toml`、`config.example.toml`、README、测试和 OpenSpec 变更 `codex-style-permission-profiles`。

## 2026-07-06 (audit)

- 审计日志默认位置从 `<cwd>/audit.jsonl` 改为 `~/.pi/agent/extensions/pi-perm/audit.jsonl`，不再在项目工作目录创建 `audit.jsonl`。
- `audit.file` 仍为相对路径配置项；绝对路径和逃出 `runtime.baseDir` 的路径在配置加载阶段被拒绝，fail-closed。
- `core/audit.ts:auditEvent` 签名从 `(config, event, cwd)` 改为 `(config, event, auditFile)`，运行时不依赖 cwd。
- `state.auditFile` 在 `createPiPermExtension` 中通过 `resolveAuditFile(config, runtimeBaseDir)` 解析一次，后续所有 `auditEvent` 调用复用同一个绝对路径。
- `defaults/base.toml` 中 `audit.file = "audit.jsonl"` 保持不变（含义自动变为 runtime.baseDir 下），无需用户改配置。
- 同步更新 `README.zh-CN.md`、`doc/PiPerm实现文档.md`、`config.example.toml`。
- **BREAKING**：旧用户依赖 `audit.jsonl` 写入项目目录的需手动迁移历史日志。

## 2026-07-06 (read-only allowlist)

- 新增 bash 只读命令白名单快速通道：`ls`、`cat`、`head`、`tail`、`wc`、`grep`、`rg`、`find`（不含 `-delete` / `-exec` / `-execdir` / `-ok` / `-okdir`）、`stat`、`file`、`tree` 对当前工作目录内路径的访问默认放行，不再走 `tools.bash.operations` 规则。
- 新增 `[tools.bash].readOnlyCommands` 配置项，用户可追加自定义只读命令（例如 `bat`、`fd`、`ag`）。
- 统一 file 工具与 bash 工具的路径策略：`read` 工具对当前工作目录内路径默认放行，cwd 外路径仍按 `allowRead` 匹配；`denyRead` 始终优先。
- 简化默认配置：删除 `strict` profile，保留 `workspace` 作为唯一内置 profile；`activeProfile` 默认 `workspace`。
- 调整 `defaults/base.toml`：`workspace` profile 的 `allowRead` 包含 `"."`；`[tools.bash]` 默认 `readOnlyCommands = []`。
- 遇到变量、命令替换、管道链中存在非只读命令等复杂情况时，bash 只读快速通道保守回退到 `tools.bash.operations` 规则，不依赖 shell 解析库。
- 补充测试覆盖：内置白名单、`isReadOnlyCommand` 写标志过滤、`extractReadOnlyPaths` 保守解析、`isPathInsideCwd` 路径范围、`evaluateBashReadAccess` 全场景、`evaluateFileAccess` read 工具 cwd 内放行、用户追加白名单、复杂命令回退。
- 同步更新 `README.zh-CN.md`、`doc/PiPerm实现文档.md`、`config.example.toml`。

## 2026-07-04

- SRT settings 默认改为写入 pi-perm extension 数据目录 `~/.pi/agent/extensions/pi-perm/runtime/`，不再在当前项目目录创建或使用 `runtime/`。
- `runtime.settingsDir` 只允许相对路径，并解析到 `runtime.baseDir` 下；绝对 `settingsDir` 会被配置校验拒绝。
- 新增 `runtime.sessionAllowTtlMs`，session 级授权会按空闲 TTL 自动清理，过期后重新请求确认。
- 新增确认选项：命中 `confirm` 规则时支持“拒绝 / 允许一次 / 本 session 始终允许”。
- `本 session 始终允许` 只在当前 Pi session 内通过内存缓存生效，不写入配置；授权范围按 profile、工具、规则或操作 ID、目标摘要限定。
- 切换 profile 时清空 session 授权缓存，避免旧 profile 授权影响新 profile。
- 补充 session 授权测试，覆盖重复命中跳过确认、不同目标仍需确认、profile 切换缓存清理和旧 `ctx.ui.confirm` 兼容行为。
- 等待用户处理权限确认期间，通过 Pi UI 状态 API 显示 blocked 风格状态，并在确认完成或异常后恢复默认 working 状态。
- 已安装 Herdr Pi integration 时，等待权限确认期间通过 Pi event bus 发送 `herdr:blocked` active true/false，让 Herdr 可显示 blocked 状态。
- 明确 Herdr `done` 由 idle/unseen 派生，pi-perm 释放 blocked 状态后交由 Herdr 自行显示 done 或 idle。
